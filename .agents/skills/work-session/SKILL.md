# Выполнение рабочей задачи (work-session)

Когда пользователь ставит рабочую задачу в рамках проекта (или вызывает этот навык), строго следуй этим правилам:

## 1. Источники и контекст
Перед выполнением задачи, если требуется дополнительный контекст, сверяйся с:
- `AGENTS.md` — основные правила проекта
- `index.md` — карта проекта (обязательно прочитай первым!)
- `docs/wiki/PROJECT_CONTEXT.md` — контекст и логика
- Исходным кодом в `src/lib/` (используй `src/lib/index.md` для навигации)
Используй другие подходящие навыки по мере необходимости (например, `brainstorming`, `frontend-design`, `systematic-debugging`).

## 2. Ограничения (Строго соблюдать!)
- Не выдумывай структуру данных и бизнес-логику — обязательно спрашивай, если не уверен.
- Следуй архитектуре: Riverpod 3 + go_router. Используй `freezed` для моделей.
- Не изменяй `Firestore rules` и `pubspec.yaml` без явного разрешения.

## 3. Отчёт о выполнении
После завершения задачи, всегда предоставляй отчет в следующем формате:
- **Что было запрошено** (кратко переформулируй задачу)
- **Что было сделано** (пунктами: ✅ / ⚠️)
- **Какие файлы изменены** (список измененных файлов)
- **Lessons Learned (Выводы)**: Опиши 1-2 урока (ошибки, инсайты), которые ты извлек из этой сессии, чтобы будущие агенты не повторяли их. Запиши это в лог сессии.
- **Команды для проверки**:
  ```bash
  cd src
  flutter analyze
  flutter test
  dart run build_runner build --delete-conflicting-outputs
  ```
- **Следующий шаг** (одно предложение о том, что дальше)
