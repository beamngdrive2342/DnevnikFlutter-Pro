# Фаза 3 — Разрезать файлы-монстры

**Роль:** опытный Flutter-разработчик. Проект `dnevnik_app`. Рабочая директория `src/`.
**Предусловие:** Фазы 1 и 2 закрыты (есть тесты-страховка и состояние на Riverpod).

**Цель фазы:** привести крупные экраны (800–1000 строк) к поддерживаемому виду: отделить
бизнес-логику/сеть от отрисовки, вынести повторно используемые куски UI в отдельные виджеты.
Никаких изменений поведения и внешнего вида — только структура.

**Риск:** средний (много перемещений кода). Страховка — тесты фазы 1 + ручной прогон каждого экрана.

---

## Definition of Done

1. Ни один файл в `lib/screens` и `lib/widgets` не превышает ~400 строк (ориентир, не догма;
   допустимы обоснованные исключения, зафиксированные в `NOTES.md`).
2. Бизнес-логика/сетевые вызовы из экранов вынесены в провайдеры/сервисы (не в `build`).
3. Крупные поддеревья UI вынесены в отдельные приватные виджеты (в идеале — отдельные файлы),
   а не в методы `Widget _buildXxx()` на 150+ строк.
4. Поведение и вид приложения не изменились. `flutter analyze` чистый, `flutter test` зелёный.

## Метод рефакторинга (одинаковый для каждого файла)

1. **Прочитать файл целиком**, составить список ответственностей (что он делает).
2. **Выделить бизнес-логику** (валидация, сетевые вызовы, преобразования) в:
   - методы соответствующего Riverpod-нотифаера (если это про домашку/расписание/auth), или
   - отдельный вспомогательный класс/контроллер, или
   - чистые функции в `lib/utils/` (если это преобразование без состояния).
3. **Выделить UI-блоки** в отдельные виджеты:
   - крупные `_buildSection()` → `class _XxxSection extends StatelessWidget/ConsumerWidget`;
   - выносить в отдельные файлы в подпапку рядом (пример уже есть: `lib/screens/diary/diary_top_bar.dart`,
     `lib/widgets/diary/*`). Держаться этого сложившегося паттерна.
4. **Убрать дублирование** (одинаковые декорации/кнопки/паддинги → общий виджет или тема).
5. **Прогнать** analyze + test + ручной сценарий экрана. Только потом коммит.

Файл-ориентир по декомпозиции для `add_homework_modal.dart`:
- `AddHomeworkController`/провайдер: распознавание через `AIService`, сбор `HomeworkItem`,
  отправка через `homeworkProvider.notifier`.
- Виджеты: поле ввода задания, выбор предмета, выбор даты (date picker), блок превью
  прикреплённых фото, кнопка «распознать ИИ», кнопка сохранить.
- Экран остаётся тонкой компоновкой этих виджетов + подписка на состояние контроллера.

## Ограничения scope
- Не менять внешний вид, тексты, анимации, тайминги.
- Не менять сетевые контракты.
- Не переходить на Firebase SDK (фаза 4).
- Не «улучшать заодно» логику — если увидел баг, записать в `NOTES.md`, не чинить в этой фазе
  (иначе рефактор структуры смешается с изменением поведения и станет неревьюибельным).

## Верификация

```bash
cd src
flutter analyze
flutter test
```
Для **каждого** переработанного экрана прогнать ручной сценарий (`flutter run`):
- `add_homework_modal`: открыть модалку → ввести текст → распознать ИИ → приложить фото →
  сохранить → ДЗ появилось в дневнике.
- `create_class_screen`: пройти мастер создания класса до конца (можно на тестовых данных).
- `class_settings_tab`: изменить расписание → сохранить → изменения видны в дневнике.
- `settings_screen`: смена имени/аватара; открыть (не подтверждать) диалог удаления аккаунта.
- `admin_panel_screen`: переключение вкладок, список ДЗ отображается.

Показать: список вынесенных виджетов/контроллеров, размеры файлов до/после, вывод analyze/test.
