# Фаза 2 — Убрать глобальную статику, перевести состояние на Riverpod

**Роль:** опытный Flutter-разработчик. Проект `dnevnik_app`. Рабочая директория `src/`.
**Предусловие:** Фаза 1 закрыта (есть зелёный `flutter test` как страховка). Если тестов нет —
СНАЧАЛА сделать фазу 1.

**Цель фазы:** устранить глобальное мутабельное статическое состояние и сделать данные
реактивными через уже подключённый `flutter_riverpod`. Сейчас всё приложение держит состояние
в статических полях (`AuthService._idToken`, `FirestoreService._classId`/кэш,
`ClassSchedule._schedule/_subjects/...`), а UI дёргает статику напрямую в обход Riverpod.

**Риск:** средний. Меняем «нервную систему» приложения. Двигаемся маленькими шагами,
после каждого — `flutter test` + ручной прогон.

---

## Definition of Done

1. `ClassSchedule` больше не источник правды в UI: расписание/предметы/времена читаются через
   Riverpod-провайдер; глобальные геттеры `weekSchedule`/`allSubjects`/`lessonTimes` либо удалены,
   либо оставлены только как тонкая обёртка над провайдером и не используются в новых местах.
2. Токен авторизации и `classId` живут в состоянии провайдеров, а не в статических полях сервисов.
3. `DiaryScreen` — `ConsumerStatefulWidget`, домашку получает из провайдера (реактивно),
   без прямых вызовов `FirestoreService.peekHomeworkCache()` / ручной перекачки `setState`.
4. Переключение аккаунта/логаут не оставляет «протёкшего» состояния прошлого пользователя.
5. `flutter analyze` чистый, `flutter test` зелёный, тесты фазы 1 адаптированы (не удалены ради галочки).

---

## Важный контекст / подводные камни

- `AuthService` и `FirestoreService` — статические и переплетены: `FirestoreService._headers()`
  читает `AuthService.idToken`. Нельзя развязать одно без другого.
- Много кода по всему приложению обращается к глобальным геттерам из `schedule_data.dart`
  (`weekSchedule`, `allSubjects`, `lessonTimes`). Замена — широкая, поэтому делаем через
  совместимость (см. стратегию), а не «большим взрывом».
- Кэш домашки уже имеет продуманную стратегию stale-while-revalidate — её **поведение сохранить**,
  просто перенести владельца состояния в Notifier.

## Стратегия (обязательна к соблюдению)
Идём **инкрементально с временными фасадами**, чтобы приложение оставалось рабочим после
каждого коммита:
1. Ввести провайдеры и перенести в них состояние.
2. Старые статические классы временно оставить как **тонкие фасады**, делегирующие в провайдеры
   (или переходно читающие из них), чтобы не переписывать все экраны разом.
3. По одному переключать экраны на `ref.watch`.
4. Удалить фасады, когда не осталось потребителей.

## Шаги

### Шаг 1. Слой сессии/сети как инстанс + провайдер
- Превратить `AuthService` и `FirestoreService` из «всё static» в обычные классы с состоянием
  экземпляра (`idToken`, `classId`, кэш — поля экземпляра, не static).
- Общий HTTP-клиент и генерацию заголовков вынести так, чтобы `FirestoreService` получал токен
  из инстанса `AuthService` (через конструктор/ссылку), а не из статики.
- Зарегистрировать провайдеры:
  - `authServiceProvider` (Provider<AuthService>),
  - `firestoreServiceProvider` (Provider<FirestoreService>, зависит от authService).
- На этом шаге можно оставить статические фасады, делегирующие в singletons провайдеров, чтобы
  не сломать существующие экраны. Цель шага — развязать владение состоянием.
- Тесты фазы 1, завязанные на статику Firestore-парсинга, перенести на инстансные/top-level
  функции (парсинг документа — чистая функция, от инстанса не зависит).
- Коммит: `refactor(data): AuthService/FirestoreService как инстансы за провайдерами`.

### Шаг 2. Auth-состояние
- `auth_provider.dart` уже есть (`AuthNotifier`/`AuthState`) — переключить его на новые
  инстансные сервисы через `ref.read(authServiceProvider)` вместо статических вызовов.
- Гарантировать, что `logout()`/`deleteAccount()` полностью очищают состояние (токен, classId,
  кэш домашки, `ClassSchedule`). Добавить/поправить тест: после logout провайдеры в исходном состоянии.
- Коммит: `refactor(auth): AuthNotifier через провайдеры сервисов`.

### Шаг 3. Расписание из статики в Notifier
- Создать `scheduleProvider` (`Notifier<ClassScheduleState>` или `NotifierProvider`), где
  `ClassScheduleState` — иммутабельная модель: `className`, `schoolName`, `classCode`, `classId`,
  `schedule (Map<int,List<Lesson>>)`, `subjects`, `lessonTimes`, `isLoaded`.
- Логику `loadFromFirestoreDoc` перенести в метод нотифаера (парсинг документа можно оставить
  как чистую top-level функцию `parseClassDoc(Map) -> ClassScheduleState`, её же покрыть тестом).
- Дефолты (`_defaultWeekSchedule`, `defaultSubjects`, `defaultLessonTimes`) оставить как есть,
  но выдавать их из нотифаера, когда `!isLoaded`.
- Глобальные геттеры `weekSchedule`/`allSubjects`/`lessonTimes` НЕ удалять сразу: временно
  переключить их на чтение из провайдера через глобальный `container`/`ref` НЕЛЬЗЯ — вместо этого
  экраны переводить на `ref.watch(scheduleProvider)`. Геттеры оставить только для мест, куда ещё
  не дошли руки, и вынести в `NOTES.md` список оставшихся потребителей.
- Коммит: `feat(schedule): scheduleProvider вместо статического ClassSchedule`.

### Шаг 4. Домашка как AsyncNotifier
- Создать `homeworkProvider` (`AsyncNotifier<List<HomeworkItem>>` или
  `AsyncNotifierProvider`), который владеет: загрузкой, кэшем в памяти, TTL, дисковым кэшем,
  фоновым refresh, оптимистичными апдейтами (`add/update/delete/setDone`).
- **Сохранить поведение** stale-while-revalidate из текущего `FirestoreService`:
  сначала отдать кэш/диск, в фоне обновить сеть, затем эмитить свежие данные.
- Методы мутаций (`addHomework`, `updateHomework`, `deleteHomework`, `setHomeworkDone`)
  обновляют состояние нотифаера, а не статический кэш.
- `HomeworkItem.done`: сейчас мутируется по ссылке внутри «unmodifiable» списка — при переносе
  сделать корректно (создавать новый объект через `copyWith`/новый инстанс), список — иммутабельный.
- Коммит: `feat(homework): homeworkProvider (AsyncNotifier) со stale-while-revalidate`.

### Шаг 5. Переключить DiaryScreen на провайдеры
- `DiaryScreen`: `StatefulWidget` → `ConsumerStatefulWidget`.
- Убрать `FirestoreService.peekHomeworkCache()` и ручные `_loadCustomHomework`/`reloadHomework`
  на `setState` — вместо этого `ref.watch(homeworkProvider)` и построение `_homeworkLookup`
  из данных провайдера (мемоизировать через `ref.listen`/`select`, чтобы не пересобирать зря).
- Pull-to-refresh дергает метод провайдера (`ref.read(homeworkProvider.notifier).refresh()`).
- Скелетоны показываются на `AsyncValue.loading`, ошибки — на `AsyncValue.error`.
- Расписание в экране брать из `ref.watch(scheduleProvider)`, не из глобального `weekSchedule`.
- Коммит: `refactor(diary): DiaryScreen на ConsumerStatefulWidget + провайдеры`.

### Шаг 6. Дочистить фасады
- Найти оставшихся потребителей статических геттеров/сервисов (grep) и перевести на `ref.watch`/`ref.read`.
- Удалить статические фасады `AuthService`/`FirestoreService`/`ClassSchedule`, когда потребителей не осталось.
- Если какой-то экран (напр. admin_panel, settings, create_class) не успеваем — оставить его фасад,
  явно записать остаток в `NOTES.md`. Фаза считается закрытой, только если DiaryScreen + auth-поток
  полностью на провайдерах; остальные экраны допустимо переключать здесь же или в фазе 3.
- Коммит: `refactor: удалить статические фасады состояния`.

## Ограничения scope
- Не менять сетевой контракт с Firestore/Vercel (формат запросов/ответов тот же).
- Не переходить на Firebase SDK (это фаза 4).
- Не заниматься дроблением файлов ради дробления (фаза 3), кроме того, что естественно возникает.
- UX-поведение кэша сохранить 1:1 (никакого «мигания» скелетонов там, где его не было).

## Верификация

```bash
cd src
flutter analyze
flutter test
```
Ручные сценарии (прогнать реально, `flutter run`):
1. **Холодный старт залогиненного пользователя:** домашка появляется без мигания скелетонов
   (кэш применился), затем тихо обновляется из сети.
2. **Pull-to-refresh** в дневнике перезагружает домашку.
3. **Добавление ДЗ** (админ) → карточка появляется в дневнике без полной перезагрузки экрана.
4. **Отметка «выполнено»** сохраняется и переживает переход между днями/перезапуск.
5. **Logout → вход под другим классом:** нет данных прошлого класса (проверка на «протёкшую» статику).

Показать вывод `flutter test` и краткий отчёт: какие провайдеры введены, какие экраны
переключены, что (если что-то) осталось на фасадах и записано в `NOTES.md`.
